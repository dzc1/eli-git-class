Git-Commit-to-Changelog Pipeline

Automatically generate release notes and changelogs by parsing Git commit history using the Conventional Commits standard, grouping changes by type.

Skill Name: changelog-automator Priority: P1 Timeline: Month 2 (Phase 1)

How It Works

  1. Skill monitors Git repositories for new commits and PR merges
  2. Parses commit messages following the Conventional Commits standard
  3. Groups changes by type (features, fixes, breaking changes, deprecations, etc.)
  4. Uses AI to generate human-readable release notes and summaries
  5. Publishes changelog to Confluence and GitHub releases

Conventional Commits Standard

The skill requires commits to follow the Conventional Commits format for reliable parsing:

Format

<type>(<scope>): <description>

Examples

# Good ‚Äî feature addition
feat(booking): add multi-currency support for EUR and GBP

# Good ‚Äî bug fix
fix(api): resolve timeout in availability endpoint

# Good ‚Äî documentation update
docs(readme): update API authentication examples

# Good ‚Äî performance improvement
perf(search): optimize hotel search query for large result sets

# Bad ‚Äî too vague
fixed stuff
update code
changes

Supported Commit Types

  • feat ‚Äî New feature (appears in "New Features" section)
  • fix ‚Äî Bug fix (appears in "Bug Fixes" section)
  • docs ‚Äî Documentation changes (optional in changelog)
  • style ‚Äî Code style/formatting (hidden by default)
  • refactor ‚Äî Code refactoring (optional in changelog)
  • test ‚Äî Test changes (hidden by default)
  • chore ‚Äî Build/dependency (optional, aggregated)
  • perf ‚Äî Performance (appears as special section)

Breaking Changes

To mark a breaking change, add BREAKING CHANGE: in the commit footer:

feat(api): redesign search endpoint response schema

BREAKING CHANGE: /search endpoint now returns paginated results with limit/offset instead of 'more' flag

Tooling Integration

  • Commit Parser: Commitlint or custom parser validates Conventional Commits format
  • Version Detection: Git tagging with semantic versioning for release identification
  • Ticket Linking: Jira REST API to extract and link ticket IDs from commit messages
  • Notifications: Slack API and email service for stakeholder alerts
  • Publication: GitHub Releases API and Confluence REST API for multi-channel publishing

Example Workflow ‚Äî Release v2.1.0

  1. Engineer pushes Git tag v2.1.0 to repository
  2. Webhook triggers the changelog-automator skill
  3. Skill queries all commits between v2.0.0 and v2.1.0
  4. Parses commit messages and extracts type, scope, and description
  5. Groups commits by type (feat, fix, perf, etc.)
  6. Generates AI-enhanced human-readable descriptions
  7. Creates CHANGELOG.md file and commits to repository
  8. Publishes to GitHub Releases with formatted notes
  9. Posts to Confluence in "Releases" space
  10. Sends Slack notification to #releases channel
  11. Notifies support/product teams with distribution links

Governance & Quality

The skill enforces commit standards through:

  • Pre-commit Hooks: Validate commit format before push
  • CI/CD Checks: Fail PR merge if commits don't follow standard
  • Human Review: Generated changelog reviewed by technical writer before publication
  • Confidence Scoring: AI-generated summaries flagged if confidence < 90%
  • Audit Trail: All generated changelogs tracked with version and timestamp
