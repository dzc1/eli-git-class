Vision: Documentation as Intelligent Infrastructure

The Problem ‚Äî Documentation as an Afterthought

Traditional documentation workflows at WebBeds face five critical structural challenges:

  • Reactive, not proactive: Documentation created after features ship, often incomplete or outdated
  • Manual bottleneck: Technical writers depend on engineering handoffs, tribal knowledge, and manual processes
  • Single point of failure: Knowledge trapped in individuals, not captured systematically
  • Version drift: Code evolves daily; documentation reviewed quarterly at best
  • Inconsistent quality: No standardised structure, formatting, or validation across the organisation
Current State Reality

Documentation lags releases by weeks or months ¬∑ API changes can break integration guides without detection ¬∑ Release notes manually compiled from scattered Jira tickets ¬∑ 27% of Confluence pages not updated in 5+ years

The Paradigm Shift ‚Äî Documentation as Code

The proposal treats documentation as intelligent infrastructure that operates on four principles:

1. Generates from source. Documentation is derived from code, not written separately. API schemas become API reference pages. Docstrings become SDK documentation. Git commits become changelogs. Database migrations become schema documentation. Agentic skills scan repositories, extract metadata, and generate human-readable content using AI-powered natural language generation.

2. Lives in version control. Documentation follows the software development lifecycle ‚Äî stored in Git repositories alongside code, branched and versioned with releases, reviewed through pull requests, and deployed automatically on merge. Skills monitor repository changes, trigger documentation updates, and publish versioned artefacts to Confluence and external portals.

3. Validates continuously. Documentation quality is enforced through automated checks: broken link detection, code example validation, metadata completeness, and freshness enforcement. Governance skills audit generated documentation, flag issues, and create remediation tickets automatically.

4. Educates the organisation. Technical writers become documentation engineers. They learn Git workflows, use AI-assisted authoring tools, configure and monitor agentic skills, and focus on high-value editorial work rather than manual page creation.

Risks & Mitigation

AI Hallucinations in Generated Documentation
Critical ‚Äî High Impact

Customers integrate based on wrong documentation, causing production incidents.

  • Validation layer: Every AI-generated doc validated against source (OpenAPI spec, actual code)
  • Confidence scoring: AI outputs a score (0‚Äì100%); below 90% triggers mandatory human review
  • Human-in-the-loop: Technical writers review all generated docs before first publish
  • Automated testing: Generated code examples run in CI/CD to verify they execute correctly
  • Feedback loop: Support tickets mentioning docs are tracked to flag problematic pages
Responsible: Documentation Infrastructure Engineer + Technical Writing Lead
Incorrect or Outdated Auto-Generated Text
Critical ‚Äî High Impact
  • Version pinning: Skills generate docs from specific Git tags, not moving branches
  • Staleness detection: Governance skill flags docs not refreshed within SLA threshold
  • Change detection: Source code changes trigger automatic documentation regeneration
  • Diff reviews: Human reviewers see a clear diff of what changed between doc versions
Responsible: Documentation Infrastructure Engineer
Sensitive Information Leakage
Critical ‚Äî Security Breach Risk
  • Secret scanning: Pre-publish validation scans for API keys, passwords, internal URLs
  • Redaction lists: Maintained list of sensitive terms filtered before publish
  • Access controls: Generated docs published to internal Confluence first; manually promoted to public
  • Audit trails: Full log of all publications ‚Äî who approved, what was published, when
  • Engineer awareness: Guidelines on avoiding sensitive content in annotations covered in engineer training programme (cross-referenced with Section 5.2)
Responsible: Security Team + Technical Writing Lead
Engineers Reject New Workflow
High Impact

Without quality code comments, AI-generated documentation will be poor quality.

  • Incentives: Teams with best documentation recognised in engineering reviews
  • Tooling: GitHub Copilot reduces friction for writing docstrings
  • Enforcement: Code review checkpoints require docstrings for all new functions
  • Showcase value: Demonstrate how good docstrings reduce the team's own support burden
  • Gradual rollout: Start with willing pilot teams; expand based on demonstrated success
Responsible: Engineering Leadership + Technical Writing Lead
Over-Reliance on Automation
Medium Impact
  • Hybrid model: Writers maintain manual documentation capability for conceptual guides and tutorials
  • Fallback procedures: Documented manual processes activated if automation infrastructure fails
  • Skills preservation: Training programme includes manual authoring best practices
  • Redundancy: All critical documentation backed up in Git
Responsible: Technical Writing Lead
Documentation Bloat
Medium Impact
  • Public vs private filtering: Only public APIs documented by default; internal utility functions excluded
  • Relevance scoring: AI scores document relevance; only high-value content published
  • Hierarchical structure: Organised by use case, not by code structure
  • Search optimisation: Investment in Confluence search improvements and tagging strategy alongside generation rollout
Responsible: Technical Writing Lead
